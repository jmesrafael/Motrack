# EXPORT_IMPORT.md — User-Facing Export (CSV/PDF)

> **Owns:** CSV exports and the PDF service-history report (R-16). Whole-database backup/restore is a different concern: [BACKUP_RECOVERY.md](BACKUP_RECOVERY.md). Entry surface: S-35. All free ([PROJECT_MISSION.md](PROJECT_MISSION.md) §3).

## 1. Principles

Exports are for **humans and spreadsheets** (accountants, buyers, personal records) — denormalized, readable, localized headers. Backup archives are for machines (lossless). Never confuse the two ([DECISION_LOG.md](DECISION_LOG.md) D-014).

## 2. CSV exports (4 domains)

Generated by `ExportService` (in-house writer, ADR-018), UTF-8 **with BOM** (Excel compatibility), CRLF, comma-separated, RFC-4180 quoting. Scope: active bike or all bikes (toggle). Filename: `motrack-<domain>-<bike|all>-<yyyyMMdd>.csv`. Delivered via share sheet (temp file in cache dir, cleaned after share).

| Domain | Columns (in order) |
|---|---|
| Maintenance | date, bike, component (localized name), service type, odometer_km, cost, brand, quantity, details (flattened `key: value; …`), notes |
| Fuel | date, bike, odometer_km, liters, total_cost, price_per_liter, station, full_tank, km_since_last_full, km_per_liter (span value or blank), notes |
| Expenses (unified view, [BUSINESS_RULES.md](BUSINESS_RULES.md) §9.1) | date, bike, category (localized), amount, source (fuel/maintenance/repair/manual), description, notes |
| Repairs | date, bike, title, odometer_km, problem, diagnosis, solution, shop, cost, notes |

Money exported as decimal pesos (`1250.00`, no ₱ symbol — spreadsheet-friendly); dates ISO `YYYY-MM-DD`; headers localized to the app language.

## 3. CSV safety

Cells beginning with `= + - @` are prefixed with `'` (formula-injection guard, [SECURITY.md](SECURITY.md) §5). Fields with commas/quotes/newlines are quoted per RFC 4180. The writer is unit-tested against hostile fixtures ([TESTING.md](TESTING.md) §4).

## 4. PDF service-history report (the resale document)

Generated via expo-print (HTML template + tokens-consistent styling, print-friendly light theme only). Per bike. Contents, in order:

1. **Header:** bike photo, nickname, brand/model/year, plate, VIN/engine no. (if set), current odometer, report date, app attribution.
2. **Summary:** total records, span (first→last date), total maintenance spend, repair spend, oil-change count ([BUSINESS_RULES.md](BUSINESS_RULES.md) §9.2).
3. **Maintenance timeline:** chronological table — date, odometer, component, service type, cost, brand; month-grouped.
4. **Repairs:** date, odometer, title, problem→solution, shop, cost.
5. **Footer each page:** "Generated by Motrack" + page n/m + generation date.

Explicitly excluded: documents/photos (privacy — a buyer needs history, not the owner's OR/CR), fuel and expense data (not resale-relevant). Health Score excluded (point-in-time number without context misleads buyers; D-015).
Filename `motrack-history-<nickname>-<yyyyMMdd>.pdf`; share sheet delivery. Long histories paginate (test with 500-record fixture; [PERFORMANCE.md](PERFORMANCE.md) §3 export budget).

## 5. Import

**MVP imports only Motrack backup archives** ([BACKUP_RECOVERY.md](BACKUP_RECOVERY.md)) — no CSV import (mapping arbitrary spreadsheets is a support tarpit; revisit on demand, D-014). The CSV format above is versioned by column order + a `# motrack-csv v1` comment line, so a future importer can read our own exports.

## 6. Evolution

**Phase 2:** QR-shared history (R-35) reuses the §4 content model server-rendered; cloud export-to-email. **Phase 3:** fleet CSV bundles (per-fleet zip), workshop invoice attachments in PDF appendix (owner-controlled). **Long-term:** verified-history PDF with server signature (resale trust product).
